# Wargame Architecture — 디지털 트윈 워게임 아키텍처 설계

리서치로 검증된 4계층 스택을 해군 국방 도메인에 맞게 구체화한다. 핵심 산출물은 **계층 간 인터페이스 계약**이다 — 이것이 빌더와 QA의 작업 기준이 된다.

## 4계층 참조 아키텍처

```
┌─────────────────────────────────────────────┐
│ L4. 디지털 트윈 상태 저장소 + 온톨로지         │  ← 실제 AIS/함대 ⇄ 시뮬 상태
│     (SQLite/Ditto, typed ontology, geo-view)  │
├─────────────────────────────────────────────┤
│ L3. LLM 지휘 계층 (CO/TAO/ASW 참모진)         │  ← 고수준 명령(의도·ROE·전력배분)
│     + HITL 가드레일 + 행동 검증 게이트         │     ※ 에스컬레이션 위험 차단 필수
├─────────────────────────────────────────────┤
│ L2. RL 전술 계층 (MAPPO self-play)            │  ← 기동·교전 행동
├─────────────────────────────────────────────┤
│ L1. 환경 코어 (PettingZoo/Gymnasium API)      │  ← reset/step, 물리·탐지·교전 판정
└─────────────────────────────────────────────┘
```

## 설계해야 할 인터페이스 계약 (필수)
빌더가 코드를 짜기 전에 다음을 **고정**한다. 각 항목은 `01_architecture.md`에 명시:

1. **디지털 트윈 상태 스키마** — 엔티티(함정/항공기/미사일/센서) 필드, 좌표계, 시간 단위. mda-gotham `vessels.db` AIS 스키마(mmsi/lat/lon/sog/cog/heading)를 입력 후보로 매핑.
2. **환경 API** — `reset(seed, scenario) -> obs`, `step(actions) -> (obs, rewards, terminations, info)`. PettingZoo `ParallelEnv` 또는 `AECEnv` 중 선택.
3. **관측/행동 공간** — 에이전트별 observation/action space 정의(부분관측 여부 포함).
4. **시나리오 스키마** — 초기 배치, 목표, ROE, 종료 조건의 구조화 포맷(JSON/YAML).
5. **에이전트 인터페이스** — `act(observation) -> action`. 규칙기반/RL/LLM 공통 시그니처.
6. **평가 메트릭 계약** — 승패 판정, 수집 지표(생존율·교전효율·시간) 정의.

## 설계 원칙 (Why)
- **인터페이스 우선, 구현 나중.** 경계를 먼저 고정하면 빌더와 QA가 병렬로 안전하게 일한다. 경계가 흔들리면 전체가 무너진다.
- **MVP부터.** 4계층 전체를 한 번에 만들면 아무것도 안 돈다. MVP = L1(PettingZoo 커스텀 env) + L4'(SQLite 상태) + 규칙기반 에이전트로 **1판 완주**. 이후 L2(RL), L3(LLM)을 증분 추가.
- **결정적 재현성.** 모든 시뮬은 시드로 재현 가능해야 한다 — 디버깅·평가·신뢰의 전제.
- **근거 각주.** 각 결정에 리서치 레퍼런스를 각주로 단다(`[ref: PettingZoo]`). 검증된 패턴 위에 서는 것이 본 하네스의 목적.

## 국방 도메인 안전 제약 (필수 명시)
- 본 시뮬레이션은 **비공식 연구·교육 목적의 합성(가상) 환경**이다. 실제 무기체계 성능 파라미터가 아닌 **공개·합성 데이터**를 전제로 설계하고, 이 가정을 설계서 상단에 명시한다.
- L3 LLM 지휘 계층은 반드시 **HITL(인간 개입) 가드레일과 행동 검증 게이트** 뒤에 둔다 — LLM의 예측 불가 에스컬레이션(Rivera et al. FAccT 2024)을 구조적으로 차단.

## 출력 형식
`_workspace/01_architecture.md`:
- 상단: 도메인 안전 가정 명시
- 4계층 다이어그램(머메이드 또는 텍스트)
- 6개 인터페이스 계약 (스키마·시그니처)
- 기술 스택 선정 + 근거 각주
- MVP → 확장 마일스톤 분할 (각 마일스톤의 "완료 정의")

설계 초안 후 `qa-verifier`에게 경계면 일관성 리뷰를 요청한다(생성-검증).
