# Foundry Research — 레퍼런스 탐색·랭킹 방법론

검증된 외부 레퍼런스 위에서 시스템을 설계하기 위해, 인기·권위 있는 자료를 정량적으로 선별한다.

## 언제 이 스킬을 쓰는가
- 새 도메인의 오픈소스/논문 지형을 파악할 때
- 아키텍트가 "이 패턴이 검증됐는가"의 근거를 요구할 때
- 사용자가 "더 최신/인기 자료로 보강"을 요청할 때

## 탐색 카테고리 (디지털 트윈 워게임 기준)
1. 멀티에이전트 시뮬/RL 환경 (PettingZoo, Gymnasium, OpenSpiel, PufferLib …)
2. 군사·해군 워게임/전술 의사결정 (AlphaStar류 self-play, DARPA ACE, CICERO …)
3. 디지털 트윈 프레임워크 (Eclipse Ditto, Azure DTDL, CesiumJS, OpenUSD …)
4. AI 워게이밍 안전성 논문·리포트 (RAND, FAccT 에스컬레이션 연구 …)
5. LLM 에이전트 오케스트레이션 (AutoGen, CrewAI, Generative Agents …)

## 랭킹 방법
- **GitHub repo:** 스타 수가 1차 지표. `https://github.com/search?q=...&s=stars` 또는 GitHub API로 확인. 최근 커밋 활성도, 포크 수, 이슈 응답성을 보조 지표로.
- **논문:** 인용수(Google Scholar/Semantic Scholar), 발표 venue(NeurIPS/ICML/FAccT/Nature 등), 발표 연도. 최신성과 인용수의 균형을 본다.
- **싱크탱크 리포트(RAND 등):** 발행기관 권위 + 인용/언급 빈도.

## 작성 규칙 (Why)
- **수치는 검증 가능해야 한다 — 지어내지 않는다.** 라이브로 못 가져오면 "추정/미확인"을 명시한다. 근거 없는 숫자는 아키텍트를 잘못된 결정으로 이끈다.
- **관련성 1줄 정당화.** "스타가 많다"가 아니라 "왜 *디지털 트윈 해군 워게임*에 관련 있는가"를 적어야 아키텍트가 활용한다.
- **상충 정보는 병기.** 자료마다 수치가 다르면 둘 다 출처와 함께 남긴다(삭제 금지).
- **URL 필수.** 모든 항목에 출처 링크.

## 권한 주의
WebSearch/WebFetch 권한이 없는 환경에서는 라이브 수치 수집이 불가하다. 이 경우:
1. 훈련 지식 기반으로 작성하되 파일 상단에 "미검증" 경고를 단다.
2. 오케스트레이터에 "네트워크 권한 있는 환경에서 재실행 필요"를 보고한다.

## 출력 형식
`_workspace/00_research_references.md`에 작성:
- 카테고리별 표: `이름 | URL | ~스타/인용 | 관련성 1줄`
- 말미에 **약 400단어 합성**: "이 레퍼런스들을 종합한 가장 검증된 아키텍처 패턴 + 권장 구체 스택 + MVP 단순화 경로".

합성 섹션이 아키텍트의 직접 입력이 되므로, 추상적 나열이 아니라 **실행 가능한 스택 추천**으로 끝맺는다.
